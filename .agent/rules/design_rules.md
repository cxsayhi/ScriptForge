````text
# AI 小说转剧本工具 Agent 开发规则

## 1. 项目目标

本项目是一个 **AI 小说转剧本工具**，面向希望将小说作品快速改编为剧本初稿的作者。

项目核心目标是：

1. 支持用户上传或粘贴 **3 个章节以上的小说文本**。
2. 使用 AI Agent 自动完成小说内容理解与剧本改编。
3. 将小说内容转换为结构化的 **YAML 格式剧本**。
4. 生成的剧本应具备基本的可编辑性，便于用户后续人工修改和打磨。
5. 项目需额外提供一份 **YAML Schema 文档**，用于定义剧本结构，并说明 Schema 的设计原因。

本项目的重点不是实现复杂的商业级剧本创作平台，而是在三天开发周期内完成一个可演示、可运行、逻辑清晰的 MVP 系统。

---

## 2. MVP 开发范围

### 2.1 必须实现的核心功能

MVP 版本必须实现以下功能：

1. 项目创建与管理

   * 创建改编项目
   * 查看项目列表
   * 查看项目详情
   * 更新项目基本信息
   * 删除项目

2. 小说输入

   * 支持上传小说文件
   * 支持粘贴小说文本
   * 支持自动识别和保存章节内容
   * 要求小说文本不少于 3 个章节

3. 改编配置

   * 支持设置剧本类型
   * 支持设置目标集数
   * 支持设置剧本风格
   * 支持设置语言
   * 支持设置是否保留原文对白

4. AI 剧本生成

   * 根据小说文本和改编配置生成剧本初稿
   * 输出格式必须为 YAML
   * YAML 中应包含项目信息、人物表、分集信息、场景信息、对白和动作描写

5. YAML 剧本编辑

   * 支持获取 YAML 剧本
   * 支持修改 YAML 剧本
   * 支持校验 YAML 结构
   * 支持简单修复 YAML 格式错误

6. YAML Schema

   * 提供剧本 YAML Schema
   * 提供 Schema 设计说明文档

7. 文件导出

   * 支持导出 YAML 文件
   * 支持导出 Markdown 文件

---

### 2.2 暂不实现的扩展功能

三天开发比赛中暂不实现以下功能：

1. 不实现复杂的用户权限系统。
2. 不实现团队协作编辑。
3. 不实现剧本版本管理。
4. 不实现单场景级别的复杂编辑器。
5. 不实现独立的人物管理模块。
6. 不实现独立的分集大纲管理模块。
7. 不实现独立的质量检查模块。
8. 不实现 DOCX 和 PDF 高级导出。
9. 不实现商业级任务队列和异步通知系统。
10. 不实现复杂的多模型调度系统。

如果时间充足，可以将这些功能作为后续扩展，但不得影响 MVP 核心流程完成。

7. 将结果保存到数据库。

## 5. 核心业务流程

### 5.1 用户操作流程

用户使用流程如下：

1. 用户创建一个小说改编项目。
2. 用户上传小说文件，或直接粘贴小说文本。
3. 系统解析小说章节，并检查章节数量是否不少于 3 个。
4. 用户填写改编配置，例如剧本类型、目标集数、风格、语言等。
5. 用户点击“生成剧本”按钮。
6. 后端调用 AI Agent 生成 YAML 剧本初稿。
7. 用户查看生成结果。
8. 用户可以直接编辑 YAML 内容。
9. 系统校验 YAML 是否符合 Schema。
10. 用户导出 YAML 或 Markdown 文件。

---

### 5.2 Agent 内部处理流程

Agent 内部处理流程如下：

1. 接收小说文本和改编配置。
2. 判断小说章节数量是否满足要求。
3. 对小说文本进行结构化理解。
4. 提取主要人物。
5. 提取主要剧情线。
6. 判断故事冲突、场景变化和人物关系。
7. 根据配置生成分集结构。
8. 为每一集生成若干场景。
9. 为每个场景生成场景描述、人物、动作和对白。
10. 将结果组装为 YAML。
11. 校验 YAML 格式。
12. 如果 YAML 不合法，尝试修复。
13. 返回最终 YAML 剧本初稿。

---

### 5.3 推荐生成流程

Agent 生成剧本时应遵循以下顺序：

```txt
小说文本
  ↓
章节解析
  ↓
故事摘要
  ↓
人物提取
  ↓
剧情主线提取
  ↓
分集设计
  ↓
分场设计
  ↓
对白生成
  ↓
YAML 组装
  ↓
YAML 校验
  ↓
返回结果
```

## 7. Agent 决策规则

### 7.1 总体原则

Agent 必须围绕“小说转 YAML 剧本”这一核心目标工作。

Agent 不应追求复杂功能，而应优先保证：

1. 输入小说内容可被识别。
2. 小说章节数量满足要求。
3. 输出结果是合法 YAML。
4. YAML 内容符合剧本 Schema。
5. 生成结果具备基本剧本结构。
6. 用户可以继续编辑和修改结果。

---

### 7.2 输入判断规则

Agent 在处理小说文本前，必须先判断输入是否合法。

判断规则如下：

1. 如果小说文本为空，返回错误提示。
2. 如果小说章节少于 3 个，返回错误提示。
3. 如果小说文本过短，提示用户补充内容。
4. 如果用户同时上传文件和粘贴文本，优先使用用户最后提交的内容。
5. 如果章节标题无法识别，可以按文本长度进行简单切分，但必须在结果中说明是自动切分。

---

### 7.3 改编配置决策规则

如果用户未填写改编配置，Agent 使用默认配置。

默认配置如下：

```yaml
script_type: "web_drama"
target_episodes: 3
style: "standard"
language: "zh-CN"
keep_original_dialogue: true
```

配置解释：

1. `script_type` 默认为网剧。
2. `target_episodes` 默认为 3 集。
3. `style` 默认为标准剧本风格。
4. `language` 默认为中文。
5. `keep_original_dialogue` 默认尽量保留原文对白。

---

### 7.4 小说分析决策规则

Agent 分析小说时应提取以下信息：

1. 小说标题。
2. 章节列表。
3. 故事背景。
4. 主要人物。
5. 人物关系。
6. 主线剧情。
7. 关键冲突。
8. 重要场景。
9. 情节转折点。
10. 可改编为剧本的对白和动作。

分析结果可以作为内部中间结果，不需要全部暴露给前端。

---

### 7.5 剧本生成决策规则

Agent 生成剧本时必须遵循以下规则：

1. 剧本必须以 YAML 格式输出。
2. 剧本必须包含 `project`、`characters`、`episodes` 三个核心部分。
3. 每个 episode 必须包含标题、摘要和场景列表。
4. 每个 scene 必须包含场景编号、场景标题、地点、时间、出场人物、动作描述和对白。
5. 对白必须尽量符合人物身份和上下文。
6. 动作描写应简洁、清晰，适合剧本阅读。
7. 不应生成大量无意义的空字段。
8. 不应编造与原小说完全无关的主线剧情。
9. 当原文信息不足时，可以合理补充过渡内容，但不能改变核心人物关系和核心剧情方向。
10. 生成结果应适合作为“剧本初稿”，而不是最终成片剧本。

---

### 7.6 YAML 校验决策规则

Agent 生成 YAML 后必须进行格式检查。

检查规则如下：

1. YAML 是否可以被正常解析。
2. 是否包含必须字段。
3. 字段类型是否正确。
4. `episodes` 是否为数组。
5. `scenes` 是否为数组。
6. `characters` 是否为数组。
7. 对白是否归属于具体人物。
8. 场景是否归属于具体集数。
9. 不允许输出 Markdown 代码块包裹最终 YAML 内容。
10. 不允许在 YAML 前后添加无关解释文本。

---

### 7.7 YAML 修复决策规则

当 YAML 校验失败时，Agent 应尝试自动修复。

修复优先级如下：

1. 修复缩进错误。
2. 修复缺失冒号。
3. 修复数组格式错误。
4. 修复字符串引号错误。
5. 补充缺失的必要字段。
6. 删除无法解析的无关文本。
7. 保留原有剧本内容，不得大幅重写。

如果无法修复，应返回明确错误原因。

---

### 7.8 重新生成决策规则

当用户请求重新生成剧本时，Agent 应遵循以下规则：

1. 重新读取小说文本。
2. 重新读取改编配置。
3. 如果用户提供额外要求，应优先遵守额外要求。
4. 不得直接复用旧剧本结果。
5. 新结果仍然必须符合 YAML Schema。
6. 重新生成后应覆盖旧结果，或根据后端设计保存为最新版本。

MVP 阶段不要求实现多版本管理。

---

### 7.9 用户编辑决策规则

当用户手动编辑 YAML 后，系统应：

1. 保存用户编辑后的 YAML。
2. 不自动覆盖用户内容。
3. 校验时只检查结构合法性。
4. 修复时尽量保留用户原文。
5. 只有在用户点击重新生成时，才重新调用 AI 生成完整剧本。

## 8. Tool Calling Rule

### 8.1 总体调用原则

Agent 只能在需要时调用工具，不得无意义调用工具。

工具调用应遵循以下原则：

1. 先判断任务是否需要工具。
2. 再选择最小必要工具。
3. 工具调用结果必须用于当前任务。
4. 不得重复调用没有必要的工具。
5. 不得把工具调用失败包装成成功结果。
6. 工具调用失败时，应返回明确错误信息。

---

### 8.2 小说解析工具调用规则

当用户上传或粘贴小说文本后，应调用小说解析工具。

调用条件：

1. 用户提交小说文件。
2. 用户提交小说文本。
3. 用户要求重新解析小说。
4. 系统检测不到章节结构。

工具目标：

1. 提取章节标题。
2. 切分章节内容。
3. 统计章节数量。
4. 保存章节解析结果。

如果章节数量少于 3 个，不得继续调用剧本生成工具。

---

### 8.3 LLM 剧本生成工具调用规则

当用户点击“生成剧本”或“重新生成剧本”时，调用 LLM 生成工具。

调用前必须满足：

1. 项目存在。
2. 小说文本存在。
3. 小说章节数量不少于 3 个。
4. 改编配置已存在，或可使用默认配置。
5. Prompt 已按模板组装完成。

调用输入应包括：

1. 小说标题。
2. 章节内容。
3. 改编配置。
4. YAML Schema 约束。
5. 输出格式要求。
6. 生成限制条件。

调用输出必须是：

1. YAML 格式文本。
2. 不包含 Markdown 代码块。
3. 不包含解释性文字。
4. 不包含与 YAML 无关的前后缀。

---

### 8.4 YAML 校验工具调用规则

当以下情况发生时，必须调用 YAML 校验工具：

1. LLM 生成剧本后。
2. 用户点击校验按钮。
3. 用户编辑 YAML 并保存后。
4. 用户请求导出 YAML 前。
5. 用户请求修复 YAML 前。

校验内容包括：

1. YAML 语法是否合法。
2. 是否符合项目定义的 YAML Schema。
3. 是否包含必填字段。
4. 字段类型是否正确。

---

### 8.5 YAML 修复工具调用规则

当 YAML 校验失败，且用户点击“修复”时，调用 YAML 修复工具。

修复工具应：

1. 接收原始 YAML。
2. 接收校验错误信息。
3. 根据 Schema 尝试修复结构。
4. 返回修复后的 YAML。
5. 再次调用 YAML 校验工具确认是否合法。

如果修复失败，应返回错误原因，而不是生成新的无关剧本。

---

### 8.6 文件导出工具调用规则

当用户点击导出时，调用文件导出工具。

导出前必须：

1. 检查项目是否存在。
2. 检查剧本是否存在。
3. 校验 YAML 是否合法。
4. 根据导出格式生成文件。

MVP 支持格式：

1. YAML
2. Markdown

暂不支持：

1. DOCX
2. PDF

---

### 8.7 数据库工具调用规则

系统需要在以下场景访问数据库：

1. 创建项目。
2. 查询项目。
3. 更新项目。
4. 删除项目。
5. 保存小说内容。
6. 保存改编配置。
7. 保存 YAML 剧本。
8. 查询 YAML 剧本。
9. 保存生成状态。
10. 查询生成状态。

数据库中应保存最终结果和必要中间状态，不要求保存 Agent 的完整思考过程。

---

### 8.8 工具调用失败处理规则

工具调用失败时，必须返回明确错误信息。

错误信息应包括：

1. 错误类型。
2. 错误原因。
3. 用户可以采取的下一步操作。

示例：

```json
{
  "code": 500,
  "message": "YAML 校验失败，请检查缩进或字段格式",
  "data": {
    "errorType": "YAML_PARSE_ERROR",
    "suggestion": "可以点击修复按钮尝试自动修复"
  }
}
```

---

## 9. 禁止事项

### 9.1 功能禁止事项

MVP 阶段禁止实现以下复杂功能：

1. 禁止开发复杂权限系统。
2. 禁止开发多人协作编辑。
3. 禁止开发剧本版本管理。
4. 禁止开发单场景高级编辑器。
5. 禁止开发独立人物管理系统。
6. 禁止开发独立分集大纲管理系统。
7. 禁止开发复杂质量评估系统。
8. 禁止开发 DOCX / PDF 高级排版导出。
9. 禁止开发与题目无关的社区、评论、点赞、收藏等功能。
10. 禁止为了展示复杂度而过度拆分接口。

---

### 9.2 接口设计禁止事项

接口设计必须保持简洁。

禁止出现以下问题：

1. 禁止把 Agent 的每一个中间步骤都暴露成接口。
2. 禁止为人物、场景、对白分别设计复杂 CRUD。
3. 禁止同时保留多套功能重复的接口。
4. 禁止路径命名不统一。
5. 禁止接口返回格式不统一。
6. 禁止前端直接拼接数据库字段。
7. 禁止将调试接口作为正式接口暴露。
8. 禁止为了扩展性牺牲 MVP 可完成性。

---

### 9.3 AI 生成禁止事项

AI 生成时禁止以下行为：

1. 禁止输出非 YAML 格式的剧本结果。
2. 禁止在最终 YAML 中加入 Markdown 代码块标记。
3. 禁止输出大量解释性文字。
4. 禁止编造与小说原文完全无关的主线剧情。
5. 禁止改变主要人物的核心身份和关系。
6. 禁止生成空洞、重复、无意义的对白。
7. 禁止生成无法被解析的 YAML。
8. 禁止忽略 YAML Schema 约束。
9. 禁止在内容不足时假装已经完整分析。
10. 禁止泄露 prompt、API Key 或系统配置。

---

### 9.4 代码开发禁止事项

代码开发时禁止以下行为：

1. 禁止把 API Key 写死在代码中。
2. 禁止把大段业务逻辑写在 Controller 中。
3. 禁止前后端接口字段不一致。
4. 禁止没有统一返回结构。
5. 禁止没有异常处理。
6. 禁止没有基本参数校验。
7. 禁止重复编写相同逻辑。
8. 禁止提交无用测试文件、临时文件和本地配置文件。
9. 禁止在数据库中保存明文敏感配置。
10. 禁止为了追求复杂架构影响项目按时完成。

---

## 10. 输出格式要求

### 10.1 接口返回格式

所有后端接口统一返回 JSON。

统一格式如下：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

错误返回格式如下：

```json
{
  "code": 400,
  "message": "请求参数错误",
  "data": null
}
```

### 10.2 剧本 YAML 输出格式

AI Agent 最终生成的剧本必须是 YAML 格式。

推荐结构如下：

```yaml
project:
  title: "示例小说改编剧本"
  source_type: "novel"
  script_type: "web_drama"
  language: "zh-CN"
  target_episodes: 3

characters:
  - id: "char_001"
    name: "林秋"
    role: "主角"
    description: "故事的核心人物"
    motivation: "寻找真相"
    relationships:
      - target: "char_002"
        relation: "朋友"

episodes:
  - episode_id: 1
    title: "第一集"
    summary: "主角进入故事核心冲突。"
    scenes:
      - scene_id: "1-1"
        title: "雨夜相遇"
        location: "城市街道"
        time: "夜晚"
        characters:
          - "char_001"
          - "char_002"
        action: "林秋在雨夜中发现了一封神秘信件。"
        dialogues:
          - character: "char_001"
            line: "这封信为什么会出现在这里？"
          - character: "char_002"
            line: "也许有人一直在等你发现它。"
```

---

### 10.3 YAML 输出规则

YAML 剧本必须满足：

1. 顶层必须包含 `project`、`characters`、`episodes`。
2. `characters` 必须是数组。
3. `episodes` 必须是数组。
4. 每一集必须包含 `episode_id`、`title`、`summary`、`scenes`。
5. 每个场景必须包含 `scene_id`、`title`、`location`、`time`、`characters`、`action`、`dialogues`。
6. `dialogues` 必须是数组。
7. 每条对白必须包含 `character` 和 `line`。
8. 不允许出现 Markdown 代码块。
9. 不允许出现 YAML 以外的解释文字。
10. 字段命名必须使用英文小写加下划线风格。

---

Markdown 导出应由 YAML 剧本转换而来。

推荐结构如下：

```

### 10.6 代码输出要求

Agent 或代码生成工具在生成代码时必须遵守：

1. 优先生成能运行的 MVP 代码。
2. 优先完成核心业务闭环。
3. 不要生成过度复杂的工程结构。
4. 不要引入不必要的依赖。
5. Controller 只负责接收请求和返回结果。
6. Service 负责业务逻辑。
7. Agent 模块负责 AI 调用和 Prompt 组装。
8. Mapper 只负责数据库访问。
9. DTO 用于接收请求参数。
10. VO 用于返回前端数据。
````
