# AI 小说转剧本工具需求文档

## 1. 项目背景

很多小说作者、短剧团队、影视改编创作者希望将已有小说内容快速改编为剧本，但传统改编过程需要人工阅读、提炼剧情、设计分集结构、拆分场次、编写对白，门槛较高且耗时较长。

本项目拟开发一款 **AI 小说转剧本工具**，通过大模型辅助完成小说理解、剧情规划、分集大纲生成、分场剧本生成和 YAML 结构化输出，帮助作者快速获得可编辑、可进一步打磨的剧本初稿。

## 2. 项目目标

本系统面向小说作者、短剧团队、影视改编创作者，支持将 **3 个章节以上的小说文本** 自动转换为结构化剧本初稿。

系统目标不是简单地把完整小说一次性交给大模型生成剧本，而是构建完整的 AI 辅助创作流程：

```text
小说导入 → 小说理解 → 剧情规划 → 分集大纲 → 分场剧本 → YAML 输出 → 人工编辑 → 导出
```

系统最终应支持生成符合 YAML Schema 规范的结构化剧本，便于作者继续修改、保存、导出和二次创作。

## 3. 用户角色

### 3.1 小说作者

小说作者可以上传或粘贴自己的小说内容，系统帮助其快速生成剧本初稿。

### 3.2 短剧团队

短剧团队可以根据小说内容快速拆分分集大纲和分场剧本，提高前期策划效率。

### 3.3 影视改编创作者

影视改编创作者可以基于系统生成的结构化剧本进行二次创作、人物调整、对白优化和剧情修改。

## 4. 核心业务流程

### 4.1 小说导入

用户可以通过上传文件或直接粘贴文本的方式导入小说内容。

系统需要支持以下输入方式：

- `.txt` 文件上传
- `.md` 文件上传
- 纯文本粘贴

系统导入小说后，需要完成以下处理：

- 自动识别章节
- 判断章节数量是否不少于 3 章
- 清洗无效文本
- 按章节和段落切分小说内容
- 保存原始小说文本

当小说章节数量少于 3 章时，系统应提示：

```text
小说章节数量不足，请上传至少 3 个章节的小说文本。
```

### 4.2 小说结构分析

系统不允许直接生成最终剧本，必须先完成小说理解和结构分析。

小说结构分析内容包括：

- 故事梗概
- 人物表
- 人物关系
- 关键事件
- 场景列表
- 时间线
- 主要冲突
- 章节摘要

### 4.3 改编参数设置

在生成剧本前，用户可以设置改编方向，使生成结果更符合创作目标。

改编参数包括：

- 剧本类型：短剧 / 网剧 / 电影 / 广播剧
- 集数
- 单集时长
- 风格：悬疑 / 甜宠 / 都市 / 古风 / 喜剧等
- 改编强度：忠于原著 / 适度改编 / 大幅改编
- 对白风格：口语化 / 文艺化 / 影视化
- 预算限制：少角色 / 少场景 / 可拍摄优先

### 4.4 分集大纲生成

系统根据小说分析结果和用户设置的改编参数生成分集大纲。

每一集大纲应包含：

- 集编号
- 集标题
- 核心剧情
- 出场人物
- 主要场景
- 情绪节奏
- 结尾钩子
- 对应原文章节

### 4.5 分场剧本生成

系统将每一集拆分为多个场次，并生成分场剧本内容。

每个场次应包含：

- 场次编号
- 场景地点
- 时间
- 内景 / 外景
- 出场人物
- 动作描述
- 人物对白
- 情绪目标
- 戏剧冲突
- 对应原文片段

最终生成结果必须支持 YAML 格式输出。

### 4.6 YAML Schema 校验

所有 AI 生成的 YAML 剧本必须经过 YAML Schema 校验。

校验内容包括：

- 字段是否完整
- 字段类型是否正确
- 层级结构是否正确
- 是否存在空场景
- 是否存在无人物对白
- 是否符合统一剧本结构

如果校验失败，系统应提示错误，并允许用户重新生成或自动修复。

### 4.7 剧本编辑与导出

用户可以对生成结果进行人工编辑和二次优化。

系统应支持：

- 修改人物设定
- 修改分集大纲
- 修改单场剧本
- 重新生成单场
- 优化对白
- 增强冲突
- 检查人物是否 OOC
- 导出 YAML
- 导出 Markdown
- 可选导出 DOCX / PDF

## 5. 功能需求

### 5.1 项目管理模块

项目管理模块用于管理用户创建的小说改编项目。

功能包括：

- 创建改编项目
- 获取项目列表
- 获取项目详情
- 更新项目配置
- 删除项目

### 5.2 小说文本模块

小说文本模块用于接收、解析和管理小说原文。

功能包括：

- 上传小说文件
- 粘贴小说文本
- 自动分章
- 获取小说章节列表
- 获取章节详情
- 重新解析章节

### 5.3 AI 小说分析模块

AI 小说分析模块用于理解小说内容，为后续剧本生成提供结构化依据。

功能包括：

- 生成小说分析结果
- 获取小说分析结果
- 更新小说分析结果
- 生成人物表
- 获取人物表
- 更新人物信息

### 5.4 改编参数模块

改编参数模块用于保存用户的剧本生成偏好。

功能包括：

- 保存改编参数
- 获取改编参数
- 更新改编参数

### 5.5 分集大纲模块

分集大纲模块用于根据小说分析结果和改编参数生成剧本的分集结构。

功能包括：

- 生成分集大纲
- 获取分集大纲
- 获取单集大纲
- 更新单集大纲
- 删除单集大纲

### 5.6 分场剧本模块

分场剧本模块用于将分集大纲进一步拆解为可拍摄、可编辑的具体场次。

功能包括：

- 生成整集分场剧本
- 获取整集分场剧本
- 获取单场剧本
- 更新单场剧本
- 删除单场剧本
- 重新生成单场剧本
- 优化单场对白
- 增强单场冲突

### 5.7 YAML 剧本模块

YAML 剧本模块用于生成、保存、编辑和校验最终结构化剧本。

功能包括：

- 生成完整 YAML 剧本
- 获取 YAML 剧本
- 更新 YAML 剧本
- 校验 YAML 剧本
- 修复 YAML 剧本

### 5.8 YAML Schema 模块

YAML Schema 模块用于定义和管理剧本结构规范。

功能包括：

- 获取 YAML Schema
- 校验 Schema 合法性
- 获取 Schema 设计说明

### 5.9 质量检查模块

质量检查模块用于检查 AI 生成剧本的质量和一致性。

功能包括：

- 执行剧本质量检查
- 获取质量检查结果
- 检查人物一致性
- 检查剧情连续性
- 检查对白自然度

### 5.10 文件导出模块

文件导出模块用于将剧本导出为不同格式。

功能包括：

- 导出 YAML 文件
- 导出 Markdown 文件
- 导出 DOCX 文件
- 导出 PDF 文件

## 6. 非功能需求

### 6.1 可编辑性

系统生成的剧本必须可编辑，用户应能够修改人物设定、分集大纲、单场剧本和 YAML 内容。

### 6.2 结构化要求

最终剧本不得只保存为普通文本，必须保存为结构化数据，至少包含：

- 项目信息
- 人物信息
- 分集信息
- 场次信息
- 动作描述
- 人物对白

### 6.3 安全性要求

前端不得直接调用大模型 API，不得暴露 API Key。所有 AI 请求必须通过后端统一转发。

### 6.4 可维护性要求

系统应采用前后端分离结构，Prompt、YAML Schema、接口逻辑和 AI Agent 逻辑应分层管理，避免代码耦合。

### 6.5 校验要求

所有 AI 生成的 YAML 剧本必须经过 Schema 校验。未通过校验的剧本不得保存为最终版本。

## 7. 技术需求

### 7.1 前端技术栈

前端建议使用：

```text
Vue 3
Element Plus
Vite
Pinia
Axios
Monaco Editor / CodeMirror
```

前端负责：

- 小说上传
- 改编参数配置
- 小说分析结果展示
- 分集大纲展示
- YAML 剧本编辑
- 质量检查结果展示
- 文件导出入口

### 7.2 后端技术栈

后端建议使用：

```text
Spring Boot
MyBatis / MyBatis Plus
MySQL
Redis（可选）
Knife4j / Swagger
```

后端负责：

- 项目管理
- 小说文本存储
- 小说分章解析
- AI 服务调用
- YAML 生成
- YAML Schema 校验
- 剧本版本管理
- 文件导出

### 7.3 AI 技术栈

AI 模块建议使用：

```text
OpenAI API / DeepSeek API / 通义千问 API / 本地 Qwen 模型
Prompt Engineering
JSON / YAML Structured Output
RAG（可选）
```

AI 模块建议拆分为：

- 小说解析 Agent
- 人物设定 Agent
- 剧情规划 Agent
- 分场设计 Agent
- 剧本生成 Agent
- 质量检查 Agent

### 7.4 数据格式

系统内部可以使用 JSON 传输，最终剧本必须导出为 YAML。

核心数据格式包括：

```text

EpisodeOutline JSON
SceneScript JSON
FinalScript YAML
YAML Schema
```

## 8. 系统页面流程

前端页面流程应围绕以下顺序组织：

```text
项目创建 → 小说上传 → 小说分析 → 分集大纲 → 剧本编辑 → 质量检查 → 文件导出
```

## 9. 接口需求

### 9.1 项目接口

| 功能         | 方法   | 路径                        |
| ------------ | ------ | --------------------------- |
| 创建项目     | POST   | `/api/projects`             |
| 获取项目列表 | GET    | `/api/projects`             |
| 获取项目详情 | GET    | `/api/projects/{projectId}` |
| 更新项目     | PUT    | `/api/projects/{projectId}` |
| 删除项目     | DELETE | `/api/projects/{projectId}` |

### 9.2 小说输入接口

| 功能               | 方法 | 路径                                   |
| ------------------ | ---- | -------------------------------------- |
| 上传小说文件       | POST | `/api/projects/{projectId}/novel/file` |
| 提交小说文本       | POST | `/api/projects/{projectId}/novel/text` |
| 获取小说内容与章节 | GET  | `/api/projects/{projectId}/novel`      |

说明：
系统只需要支持用户上传或粘贴 3 个章节以上的小说文本，并在后端完成章节解析。因此不再单独拆分“获取章节详情”“重新解析章节”等接口，避免接口过细。

### 9.3 AI 改编接口

| 功能         | 方法 | 路径                                          |
| ------------ | ---- | --------------------------------------------- |
| 生成剧本初稿 | POST | `/api/projects/{projectId}/script/generate`   |
| 重新生成剧本 | POST | `/api/projects/{projectId}/script/regenerate` |
| 获取生成状态 | GET  | `/api/projects/{projectId}/script/status`     |

说明：
原来的“小说分析、人物表、分集大纲、分场剧本、对白优化、冲突增强”等功能都属于 AI 改编过程中的中间步骤。对于三天开发比赛，不建议全部暴露成独立接口，而是统一收敛到“生成剧本初稿”接口中，由 Agent 在内部完成分析、提纲、分场和 YAML 生成。

### 9.4 剧本内容接口

| 功能           | 方法 | 路径                                        |
| -------------- | ---- | ------------------------------------------- |
| 获取 YAML 剧本 | GET  | `/api/projects/{projectId}/script`          |
| 更新 YAML 剧本 | PUT  | `/api/projects/{projectId}/script`          |
| 校验 YAML 剧本 | POST | `/api/projects/{projectId}/script/validate` |
| 修复 YAML 剧本 | POST | `/api/projects/{projectId}/script/repair`   |

说明：
题目要求最终产物是结构化 YAML 剧本，因此剧本接口只围绕 YAML 内容本身设计即可。用户可以在前端直接编辑 YAML，后端负责保存、校验和简单修复。

### 9.5 改编配置接口

| 功能               | 方法 | 路径                                 |
| ------------------ | ---- | ------------------------------------ |
| 保存或更新改编配置 | PUT  | `/api/projects/{projectId}/settings` |
| 获取改编配置       | GET  | `/api/projects/{projectId}/settings` |

说明：
改编配置可以包括剧本类型、目标集数、风格、语言、单集时长、是否保留原文对白等内容。
这里使用一个统一配置接口，不再拆分为多个细粒度接口。

### 9.6 YAML Schema 接口

| 功能                 | 方法 | 路径                        |
| -------------------- | ---- | --------------------------- |
| 获取 YAML Schema     | GET  | `/api/schema/script`        |
| 获取 Schema 设计说明 | GET  | `/api/schema/script/design` |

说明：
题目要求额外提供 YAML Schema 文档，并说明设计原因。因此只需要提供 Schema 内容和设计说明两个接口。
“校验 Schema 合法性”一般是开发阶段工作，不是用户核心功能，可以不作为正式业务接口。

### 9.7 导出接口

| 功能               | 方法 | 路径                                        |
| ------------------ | ---- | ------------------------------------------- |
| 导出 YAML 文件     | GET  | `/api/projects/{projectId}/export/yaml`     |
| 导出 Markdown 文件 | GET  | `/api/projects/{projectId}/export/markdown` |

说明：
题目重点是 YAML 格式剧本，因此必须支持 YAML 导出。Markdown 可以作为展示型导出。
DOCX 和 PDF 在三天比赛中实现成本较高，可以作为扩展功能，不放入核心接口。

## 10. 统一响应格式

所有后端接口必须使用统一响应格式：

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

错误响应格式示例：

```json
{
  "code": 400,
  "message": "小说章节数量不足，请上传至少 3 个章节的小说文本。",
  "data": null
}
```

## 12. 约束条件

### 12.1 生成流程约束

禁止将完整小说文本一次性发送给大模型并直接生成最终剧本。

必须按照以下流程执行：

```text
小说分析 → 剧情规划 → 分集大纲 → 分场剧本 → YAML 校验
```

### 12.2 YAML 校验约束

所有生成的 YAML 剧本必须经过 Schema 校验，未通过校验的剧本不得保存为最终版本。

### 12.3 接口命名约束

后端接口必须统一使用 `/api` 前缀。接口路径必须语义清晰，不允许出现：

```text
/test
/demo
/getData
/doSomething
/ai1
/tmp
```

### 12.4 大模型调用约束

前端不得直接调用大模型 API。所有 AI 请求必须通过后端统一转发。

### 12.5 文件存储约束

上传文件、导出文件不得放在 `src` 代码目录中，应放入独立存储目录。

### 12.6 结构化结果约束

最终剧本必须保存为结构化数据，不允许只保存普通文本。

## 13. 最小可行版本要求

MVP 阶段必须完成：

1. 创建项目
2. 上传或粘贴 3 章以上小说文本
3. 自动分章
4. AI 生成小说分析结果
5. AI 生成分集大纲
6. AI 生成分场剧本
7. 输出 YAML 格式剧本
8. YAML Schema 校验
9. 用户可编辑 YAML
10. 导出 YAML 文件

MVP 阶段可以暂缓：

1. 用户登录注册
2. 多人协作
3. 支付系统
4. 复杂 RAG
5. PDF 导出
6. 高级权限管理
